# Home Assistant OS 构建工作流使用指导书

## 一、工作流概述

### 1.1 工作流文件
- **位置**：`.github/workflows/build.yaml`
- **名称**：OS build
- **用途**：构建 Home Assistant Operating System 镜像

### 1.2 触发方式

| 触发方式 | 说明 | 适用场景 |
|---------|------|---------|
| `release: types: [published]` | GitHub Release 发布时自动触发 | 正式版本发布 |
| `workflow_dispatch` | 手动触发 | 开发测试、紧急修复 |

### 1.3 手动触发参数

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `boards` | string | 空（全部） | 指定构建目标，逗号分隔 |
| `publish` | boolean | true | 是否发布到 R2 |
| `run_tests` | boolean | true | 是否运行测试 |
| `hassio_channel` | choice | default | 发布通道（default/stable/beta/dev） |

## 二、工作流阶段详解

### 2.1 阶段总览

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌──────────────┐    ┌──────────────┐
│ prepare │───▶│  build  │───▶│  test   │───▶│ update_index │───▶│ bump_version │
└─────────┘    └─────────┘    └─────────┘    └──────────────┘    └──────────────┘
   准备阶段      构建阶段      测试阶段        更新索引阶段        版本推送阶段
```

---

### 2.2 阶段一：prepare（准备阶段）

**作用**：准备构建所需的所有元数据和资源

**输出变量**：

| 变量 | 说明 | 示例 |
|------|------|------|
| `version_dev` | 开发版本号 | `dev20260419` |
| `version_main` | 主版本号 | `14.2` |
| `version_full` | 完整版本号 | `14.2.dev20260419` |
| `channel` | 发布通道 | `stable/beta/dev` |
| `matrix` | 构建矩阵 | 平台列表 |
| `build_container_image` | 构建容器镜像 | `ghcr.io/.../haos-builder@sha256:...` |
| `publish_build` | 是否发布 | `true/false` |

**关键步骤**：

1. **检出源代码**
2. **检查是否发布**：仅主仓库可发布
3. **生成开发版本号**：`devYYYYMMDD` 或 `dev时间戳`
4. **设置版本后缀**：修改 `buildroot-external/meta`
5. **获取版本号**：从 meta 文件读取
6. **验证版本号**：Release 时验证与 Tag 一致
7. **确定发布通道**：根据事件类型和输入参数
8. **生成构建矩阵**：筛选要构建的平台
9. **构建 Docker 镜像**：构建 `haos-builder` 容器
10. **生成签名证书**：RAUC OTA 更新签名

---

### 2.3 阶段二：build（构建阶段）

**作用**：为每个目标平台构建系统镜像

**矩阵策略**：
- `fail-fast: true`（Release）→ 任一失败立即终止
- `fail-fast: false`（非 Release）→ 继续构建其他平台

**关键步骤**：

1. **检出源代码**（含子模块）
2. **设置 Python 和 AWS CLI**
3. **设置版本后缀**
4. **配置 RAUC 证书**
5. **释放磁盘空间**：删除 Android SDK、CodeQL 等
6. **恢复编译缓存**
7. **生成构建配置**：`make <board>_defconfig`
8. **覆盖发布通道**：非 stable 时修改配置
9. **执行构建**：`make`
10. **检查 Linux 配置**
11. **上传制品**：到 R2 或 GitHub Release
12. **缓存统计和保存**
13. **生成构建摘要**
14. **上传多种格式镜像**

**输出制品格式**：

| 格式 | 说明 | 适用平台 |
|------|------|---------|
| `.img.xz` | SD 卡镜像 | 所有平台 |
| `.raucb` | RAUC 更新包 | 所有平台 |
| `.ova` | 虚拟设备 | ova |
| `.qcow2.xz` | QEMU 镜像 | ova, generic-aarch64 |
| `.vmdk.zip` | VMware 镜像 | ova, generic-aarch64 |
| `.vdi.zip` | VirtualBox 镜像 | ova |
| `.vhdx.zip` | Hyper-V 镜像 | ova |

---

### 2.4 阶段三：test（测试阶段）

**作用**：验证构建产物的正确性

**触发条件**：
- Release 发布时
- 手动触发 + `run_tests=true`

**执行内容**：
- 下载 ova 镜像
- 启动 QEMU 虚拟机
- 运行集成测试
- 验证 Home Assistant 启动

---

### 2.5 阶段四：update_index（更新索引阶段）

**作用**：更新开发构建版本的 Web 下载索引

**触发条件**：
- 非 Release 事件
- 且需要发布到 R2

**执行内容**：
- 生成 `index.json`（所有版本列表）
- 生成 `{version}.json`（该版本所有镜像）
- 更新 `index.html`（Web 下载页面）
- 清除 Cloudflare CDN 缓存

**访问地址**：https://os-artifacts.home-assistant.io/index.html

---

### 2.6 阶段五：bump_version（版本推送阶段）

**作用**：更新版本服务器，通知用户有新版本

**触发条件**：
- 主仓库
- 需要发布
- 非 dev 通道，或 dev 分支的 dev 通道

**执行内容**：

1. **更新版本服务器**：`https://version.home-assistant.io/`
2. **stable 发布时**：同时更新 beta 通道
3. **stable Release 时**：更新 RPi Imager 版本列表

**版本通道关系**：

```
stable 发布 → 更新 stable + beta + RPi Imager
beta 发布   → 更新 beta
dev 构建    → 更新 dev
```

## 三、工作流运行流程

### 3.1 Release 发布流程

```
创建 GitHub Release
        ↓
    自动触发
        ↓
┌─────────────────┐
│ 1. prepare      │ 生成版本号、构建矩阵、Docker 镜像
└─────────────────┘
        ↓
┌─────────────────┐
│ 2. build (并行) │ 构建 14 个平台镜像
└─────────────────┘
        ↓
┌─────────────────┐
│ 3. test         │ 运行集成测试
└─────────────────┘
        ↓
┌─────────────────┐
│ 4. bump_version │ 更新版本服务器 + RPi Imager
└─────────────────┘
        ↓
   发布完成
```

### 3.2 开发构建流程（dev 分支）

```
手动触发或 Push
        ↓
┌─────────────────┐
│ 1. prepare      │ 生成 dev 版本号
└─────────────────┘
        ↓
┌─────────────────┐
│ 2. build (并行) │ 构建指定或全部平台
└─────────────────┘
        ↓
┌─────────────────┐
│ 3. test         │ 可选：运行测试
└─────────────────┘
        ↓
┌─────────────────┐
│ 4. update_index │ 更新 Web 下载索引
└─────────────────┘
        ↓
┌─────────────────┐
│ 5. bump_version │ 更新 dev 通道版本
└─────────────────┘
        ↓
   开发版本发布
```

### 3.3 Fork 仓库流程

```
手动触发
        ↓
┌─────────────────┐
│ 1. prepare      │ publish_build=false
└─────────────────┘
        ↓
┌─────────────────┐
│ 2. build (并行) │ 构建指定平台
└─────────────────┘
        ↓
┌─────────────────┐
│ 3. test         │ 可选：运行测试
└─────────────────┘
        ↓
  制品上传为 Artifact
  （可从 Actions 页面下载）
```

## 四、使用指南

### 4.1 正式版本发布

**步骤**：

1. 更新 `buildroot-external/meta` 中的版本号：
   ```bash
   VERSION_MAJOR=14
   VERSION_MINOR=3
   VERSION_SUFFIX=""
   ```

2. 提交并推送：
   ```bash
   git add buildroot-external/meta
   git commit -m "Bump version to 14.3"
   git push
   ```

3. 创建 GitHub Release：
   - Tag: `14.3`
   - Title: `Home Assistant OS 14.3`
   - 勾选 "This is a pre-release"（如果是 beta）

4. 等待工作流完成

**结果**：
- 镜像上传到 GitHub Release
- 版本服务器更新
- RPi Imager 更新（stable）

---

### 4.2 开发版本构建

**步骤**：

1. 进入 Actions 页面
2. 选择 "OS build" 工作流
3. 点击 "Run workflow"
4. 填写参数：
   - `boards`: 留空或指定平台（如 `rpi4_64,rpi5_64`）
   - `publish`: true（发布到 R2）
   - `run_tests`: true（运行测试）
   - `hassio_channel`: default

5. 点击 "Run workflow"

**结果**：
- 镜像上传到 R2
- Web 索引更新
- 可从 https://os-artifacts.home-assistant.io/ 下载

---

### 4.3 快速测试构建

**步骤**：

1. 手动触发工作流
2. 填写参数：
   - `boards`: `ova`（仅构建虚拟设备）
   - `publish`: false
   - `run_tests`: true
   - `hassio_channel`: dev

3. 等待构建完成
4. 从 Actions 页面下载 Artifact

**适用场景**：
- 快速验证代码修改
- 本地测试
- PR 验证

---

### 4.4 部分平台构建

**场景**：仅修改了树莓派相关代码

**步骤**：

1. 手动触发工作流
2. 填写参数：
   - `boards`: `rpi3_64,rpi4_64,rpi5_64`
   - `publish`: false
   - `run_tests`: false

**优势**：
- 节省 CI 时间（约 1 小时 vs 14 小时）
- 节省 CI 资源

## 六、常见问题

### Q1: 构建失败如何排查？

1. 查看 Actions 日志
2. 检查失败的平台和步骤
3. 常见原因：
   - 编译错误：检查代码修改
   - 磁盘空间不足：正常，会自动清理
   - 缓存问题：清除缓存后重试

### Q2: 如何跳过测试？

手动触发时设置 `run_tests: false`

### Q3: Fork 仓库如何构建？

1. Fork 仓库
2. 手动触发工作流
3. 设置 `publish: false`
4. 从 Actions 页面下载 Artifact

### Q4: 如何查看构建产物？

- **Release**：GitHub Release 页面
- **开发构建**：https://os-artifacts.home-assistant.io/
- **Fork 构建**：Actions → 对应运行 → Artifacts

### Q5: 版本号如何管理？

版本号定义在 `buildroot-external/meta`：

```bash
VERSION_MAJOR=14      # 主版本
VERSION_MINOR=2       # 次版本
VERSION_SUFFIX=""     # 后缀（Release 为空，开发构建自动填充）
```

## 九、总结

本工作流实现了：

1. **自动化构建**：Release 自动触发，无需手动干预
2. **多平台支持**：14 种硬件平台并行构建
3. **灵活配置**：手动触发可精确控制构建范围
4. **完整测试**：集成测试确保质量
5. **版本管理**：多通道版本发布
6. **生态集成**：RPi Imager 自动更新

通过本指导书，您可以：
- 正确发布正式版本
- 进行开发版本构建
- 快速测试代码修改
- 排查构建问题
